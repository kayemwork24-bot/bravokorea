## 0. 한 줄 요약

**앱(BK App)이 원본, 웹은 그 "공개본 사본"을 조회만 하는 읽기 전용 티저(미끼) 레이어.**
검색·SNS에서 유입된 사람에게 콘텐츠 일부를 보여주고 → **모든 상호작용을 앱 설치/딥링크로 전환**시키는 것이 웹의 유일한 목적이다. 실제 커뮤니티 활동(전체 읽기 + 모든 쓰기)은 전부 앱에서 일어난다.

---

## 1. 핵심 원칙 (반드시 유지)

| 원칙 | 내용 |
|---|---|
| **원본 = 앱** | 콘텐츠·유저·상호작용의 단일 진실 공급원(SSOT)은 BK 앱 백엔드. |
| **웹 = 읽기 전용 사본** | 웹은 앱 데이터를 **직접 호출하지 않는다.** CDN에 미리 복사된 공개 데이터 사본만 읽는다. |
| **동기화 = 단방향 (앱 → 웹)** | 웹에서 발생한 어떤 것도 앱으로 역류하지 않는다. 웹은 write를 하지 않는다. |
| **모든 상호작용 → 앱** | 좋아요·댓글·글쓰기·DM·"더 보기" 등 모든 액션 = 앱 설치/딥링크로 컷. |
| **은행 코어 ↔ 마케팅 웹 분리** | 뱅킹 코어 DB를 웹 트래픽에 직접 노출 금지. 보안·장애 격리 목적. |

## 2. 아키텍처 (목표 구조)

```
┌─────────────┐   (공개 승인된 글만)    ┌──────────────┐        ┌─────────────┐
│  BK App     │  ───── export ─────▶   │  CDN         │  read  │  Web        │
│  (원본/코어) │   posts.json / events  │  posts.json  │ ◀───── │ (PC/MOBWEB) │
│  뱅킹·커뮤니티│   (본문은 발췌만 포함)  │  (정적 사본)  │        │  정적 조회   │
└─────────────┘                         └──────────────┘        └─────────────┘
        ▲                                                              │
        └───────────────── 앱 설치 / 딥링크 전환 ◀──────────────────────┘
                         (좋아요·댓글·더보기·글쓰기 클릭 시)
```

**왜 이렇게 하나 (PM이 이해관계자에게 설명할 논리):**
- 뱅킹 코어 시스템과 마케팅 웹이 **물리적으로 분리** → 보안·장애 격리
- 웹 트래픽이 아무리 몰려도 **앱 백엔드에 부하 0**
- 정적 파일이라 **빠르고 저렴하고 SEO에 유리**

> ⚠️ **중요 보안 원칙:** 본문 전체(full body)는 **CDN/웹으로 절대 내보내지 않는다.** 앱이 export 단계에서 이미 **발췌(excerpt)로 잘라서** posts.json에 담는다. 즉 "웹에서 블러 처리"는 보안장치가 아니라 시각적 마감일 뿐이고, **실제 차단은 export 단계(앱 측)에서 일어난다.** (블러만 하고 DOM에 전체 본문을 넣으면 소스보기로 뚫린다 → 금지)

## 3. 웹이 하는 것 / 안 하는 것

### ✅ 웹이 하는 것 (조회만)
- 공개 **승인된** 글의 **제목 + 발췌(excerpt) + 지표(좋아요/댓글/조회 수)** 표시
- 카테고리·정렬 등 탐색 UI
- 이벤트/프로모션 랜딩 (마케팅 콘텐츠)
- SEO/OG 메타로 검색·SNS 유입 확보
- 모든 CTA를 **앱 설치/딥링크**로 연결

### ❌ 웹이 안 하는 것
- 글쓰기·댓글·좋아요·DM 등 **모든 쓰기(write) 동작**
- 본문 **전체** 노출 (발췌까지만)
- 로그인/회원 기능, 개인화

### 🔁 모든 상호작용 → 앱 전환
> "댓글 달기", "더 보기", "좋아요"를 누르면 전부 → **앱 설치/딥링크로 컷.**
> 딥링크: `https://abr.ge/27gghb` (Airbridge, 글 상세에서는 `?post_id=<id>`로 유입 기여 추적)

## 4. 데이터 계약 (목표: `posts.json` 스키마)

앱이 export하는 공개 사본. **`body`(전체 본문) 필드는 없다.** 발췌만.

```jsonc
{
  "generatedAt": "2026-07-09T03:00:00Z",
  "posts": [
    {
      "id": "p_ktr01",
      "category": "korean",              // CATEGORIES id
      "title": "제목",
      "excerpt": "본문 앞부분 발췌 (앱에서 N자/문단 단위로 truncate)",
      "author": "닉네임",
      "flag": "🇰🇷",
      "publishedAt": "2026-07-01T09:00:00Z",
      "metrics": { "likes": 128, "comments": 34, "views": 1500 },
      "hasMore": true,                    // 뒤에 잘린 내용이 더 있는지 (블러/CTA 노출 여부)
      "appLink": "https://abr.ge/27gghb?post_id=p_ktr01"
    }
  ]
}
```

- **승인/공개 필터는 앱(export)에서 끝낸다.** 웹은 받은 것만 그대로 그린다.
- 갱신 주기(예: 5~15분마다 배치 export)와 캐시 정책(CDN TTL)은 앱팀과 합의 필요.
- 이벤트는 별도 `events.json`으로 같은 방식(현재는 웹에 하드코딩됨 → 3장 참고).

## 5. ⚠️ 현재 상태(프로토타입) vs 목표 — 갭 분석

**지금 배포된 것은 "디자인/플로우 프로토타입"이다.** 목표 아키텍처와 다음 갭이 있으며, 운영 전환 시 반드시 처리해야 한다.

| 영역 | 현재 (프로토타입) | 목표 (운영) | 조치 |
|---|---|---|---|
| 데이터 소스 | `assets/seed.js`에 **하드코딩된 데모 글** | CDN `posts.json` fetch | **교체 필요 (P0)** |
| 저장 | 브라우저 `localStorage` (기기별) | 없음 (정적 조회만) | 제거 |
| 본문 | 상세에서 **전체 본문 노출** | **발췌 + 블러 + CTA** | **변경 필요 (P0)** |
| 글쓰기 | `#/new` compose 화면·작성 **동작함** | 없음 → 딥링크 | 라우트 제거/게이팅 (P0) |
| 좋아요 | 웹에서 토글되어 localStorage 저장 | 클릭 시 앱 딥링크 | 전환 처리 (P0) |
| 댓글 | 카운트만 표시 (버튼 동작 없음) | 클릭 시 앱 딥링크 | 전환 처리 (P1) |
| 공유 | 딥링크 공유 시트 (유입 추적) | 유지 | 유지 (인스톨 유도에 유용) |

> 즉, **읽기 전용 티저로 만들려면 "쓰기 동작 제거 + 본문 발췌화 + posts.json 연동" 3가지가 핵심 남은 작업.**

## 6. 🔑 미결정 사항 — 본문 노출 정책 (PM 결정 필요)

웹에서 커뮤니티 글을 **어디까지 보여줄지** 결정해야 한다. 두 안:

### 안 A) 전체 노출 후 자율 전환
- 글 전체를 보여주고, 사용자가 알아서 앱 설치하게 유도.
- 장점: 콘텐츠 만족도·SEO 텍스트량 최대.
- 단점: 웹에서 이미 다 읽어버려 **앱 전환 동기 약함.** 본문 전체가 웹에 노출됨(보안 원칙과 상충).

### 안 B) 일부만 + 나머지 블러 (권장)
- 발췌 일부만 보여주고 그 뒤를 블러 + "앱에서 계속 읽기" CTA.
- 장점: **"다음 내용 궁금증" → 앱 설치 전환율 상승 가능성 큼.** 본문 전체가 웹에 안 나감(보안 원칙 부합).
- 단점: 맥락이 끊길 수 있음 → 발췌 길이/컷 지점 튜닝 필요.

**권장: 안 B.** 단, 아래 세부 로직을 확정해야 한다.

#### 블러/컷 세부 로직 (제안)
1. **컷은 앱 export에서** — posts.json `excerpt`에 본문 앞 **첫 2~3문단(또는 300~400자)** 만 담는다. (전체 본문은 CDN에 안 올림 → 소스보기로도 안 뚫림)
2. **웹 표시** — 발췌를 렌더한 뒤 **마지막 1~2줄에 그라디언트 페이드/블러 오버레이**를 겹치고, 그 아래 **"앱에서 전체 보기 →"** 딥링크 CTA.
3. **`hasMore=false`**(짧은 글)면 블러/CTA 없이 발췌 그대로.
4. **SEO 주의:** 블러는 CSS 시각 효과일 뿐이고, DOM에는 **발췌까지만** 존재해야 한다. "전체 본문을 숨겨두고 크롤러에만 노출"하는 방식은 **클로킹(cloaking)으로 페널티 위험** → 금지.

> 결정 필요 항목: ① 발췌 길이(문단 수/글자 수), ② 컷 기준(문단 경계 vs 글자 수), ③ 블러 강도/CTA 문구.

## 8. 파일 구조 맵

```
bravokorea/
├─ index.html            # 헤더/네비/드로어 골격 + OG 메타 + 스크립트 로드
├─ HANDOVER.md           # (이 문서)
└─ assets/
   ├─ app.js             # 라우터 + 모든 화면 렌더 (피드/상세/이벤트/공유/대시보드)
   ├─ seed.js            # ⚠️ 데모 데이터(글·카테고리·이벤트) — 운영 시 posts.json으로 대체
   ├─ styles.css         # 전체 스타일 (브랜드 토큰 = :root 변수)
   ├─ visa.js            # 비자 점수 계산기 (웹 자체 마케팅 툴)
   ├─ visa-rules.js      # 비자 계산 룰 데이터
   ├─ og.png             # 소셜 미리보기 + 모바일 히어로 배너 이미지 (1200×630)
   └─ bk-logo*.svg       # 로고 (네이비/화이트)
```

### 화면(라우트) 개요 — `app.js` `render()`
- `#/` , `#/c/:cat` — 커뮤니티 피드 (전체/카테고리)
- `#/p/:id` — 글 상세 ⚠️(현재 전체 본문, 목표는 발췌+블러)
- `#/events`, `#/event/:id` — 이벤트/혜택 랜딩 + 상세
- `#/visa` — 비자 계산기 (웹 자체 콘텐츠)
- `#/new`, `#/track` — ⚠️ 프로토타입 잔재(글쓰기/추적 대시보드), 운영에선 제거/비공개 대상

> 참고: **이벤트·비자 계산기는 앱에서 동기화되는 콘텐츠가 아니라 웹 자체 마케팅 콘텐츠다.** (커뮤니티 글만 앱→웹 동기화 대상)

## 10. 오픈 이슈 / 리스크

- **미커밋 작업 존재:** 로컬 `app.js`에 소셜 공유 기능 WIP가 커밋되지 않은 상태로 남아 있음. 인수 시 이어서 마무리하거나 정리 필요.
- **데모 데이터 = 실제 아님:** 현재 글/이벤트는 샘플. 실제 승인 콘텐츠 연동 전까지 대외 홍보 주의.
- **본문 노출 정책(6장) 미결정** — 이게 정해져야 P0 상세 화면 작업 착수 가능.
- **SEO vs 게이팅 균형** — 블러를 시각효과로만 쓰고 DOM엔 발췌까지만(클로킹 금지).
- **도메인/딥링크 하드코딩** — 커스텀 도메인 전환 시 OG·canonical·딥링크 점검.

## 11. 요약 (PM이 기억할 것)

> **앱이 원본, 웹은 그 공개본 사본을 "조회만".**
> 웹은 검색·SNS 유입자에게 **미끼(제목+발췌+지표)**를 보여주고 **모든 액션을 앱으로 컷**하는 랜딩 레이어다.
> 지금은 **프로토타입**이며, 운영 전환의 핵심 3가지는 **① posts.json 연동 ② 본문 발췌+블러 ③ 쓰기 동작 제거(→딥링크)** 이다.
